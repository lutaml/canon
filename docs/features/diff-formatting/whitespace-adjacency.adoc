= Whitespace adjacency in diff reports
:toc:
:toclevels: 2

== Purpose

Canon's diff reports treat whitespace adjacent to content nodes specially so a
single underlying delta does not balloon into 3-4 misleading "content differs"
entries. Without this handling, positional alignment of children pairs an
expected whitespace-only text node against an actual content node, cascading
the misalignment downstream.

This page explains the mechanism, the contracts it preserves, and how it
interacts with the three-way whitespace classification (`:preserve` /
`:collapse` / `:strip`).

== The problem

Consider a metanorma-style HTML fragment compared as
`be_html4_equivalent_to`:

[source,html]
----
<!-- expected -->
<p id="x" class="NormRef">
  <span class="stdpublisher">ISO </span>
  <span class="stddocNumber">20483</span>
  ,
  <i>Cereals and pulses</i>
</p>

<!-- actual -->
<p id="x" class="NormRef"><span class="stdpublisher">ISO </span><span class="stddocNumber">20483</span>, <i>Cereals and pulses</i></p>
----

Both render identically in a browser — the indentation is structural HTML
formatting, not content. But before this fix, the diff report contained four
entries:

[source]
----
DIFFERENCE #1 — element_structure: <i>Cereals and pulses</i> "added"
DIFFERENCE #2 — text_content: ""  vs  "20483"        (visualised: ↵░░░░)
DIFFERENCE #3 — text_content: "20483"  vs  ","
DIFFERENCE #4 — text_content: ","  vs  "Cereals and pulses"  (visualised: ↵░░░░,↵░░░░)
----

The cascade comes from positional `zip()` alignment in
`Canon::Comparison::XmlComparatorHelpers::ChildComparison`: with the expected
side carrying extra whitespace-only text nodes and the actual side carrying
none, every child slides by one slot and gets paired against the wrong
neighbour.

== The contract: three whitespace classes

Canon classifies every text node's parent into one of three whitespace
classes (see `Canon::Comparison::WhitespaceSensitivity`):

`:preserve`:: Every whitespace character is significant. `" "` ≠ `"\n"`.
HTML defaults: `pre`, `code`, `textarea`, `script`, `style`. XML default:
none unless configured.

`:collapse`:: Presence vs. absence matters, but all whitespace _forms_ are
equivalent. `" "` == `"\n  "`. HTML defaults: `p`, `li`, `dt`, `dd`, `td`,
`th`, `h1`-`h6`, `caption`, `figcaption`, `label`, `legend`, `summary`,
`blockquote`, `address`, `button`. XML default: none unless configured.

`:strip`:: All whitespace-only text nodes are structural noise and are
silently dropped. Default for XML elements not in any list, and for HTML
elements outside the collapse / preserve sets.

== What this fix does

When `node_excluded?` runs on a candidate child during alignment, it checks
the parent's whitespace class. If the class is `:collapse` and the text node
content begins with `\n`, the node is filtered from both sides
unconditionally — regardless of whether the surrounding siblings are inline
elements (which would otherwise mark the whitespace as "significant" because
HTML renders inter-inline whitespace as a visible space).

Why drop it even between inline siblings? Because in any `:collapse` parent
the whitespace is structural pretty-print indentation, never visible in the
final rendered output. Browsers collapse `\n   ` between two `<span>`s to a
single space; canon's filter does the same at alignment time so the diff
report reflects the rendered reality rather than the source-text accident of
where the formatting newlines fall.

The filter only fires for `\n`-leading whitespace. Pure-space text nodes
(e.g. a single `" "` between two inline siblings) are kept — those represent
real inline content that the author typed, not pretty-printer indentation.

== What this fix does NOT do

* **`:preserve` parents are untouched.** Whitespace inside `<pre>`, `<code>`,
  `<textarea>`, etc. is preserved character-by-character. A whitespace-only
  text node in that context will still appear in the diff and still
  contribute to equivalence judgement.

* **Equivalence outcomes are not silently changed for `:preserve`.** If two
  `:preserve` parents differ in whitespace presence, that difference is
  still flagged as NORMATIVE.

* **`:strip` is unaffected.** Whitespace-only text nodes were already
  filtered out in `:strip` contexts; this fix changes nothing there.

* **Pretty-print flag interaction.** The
  `pretty_printed_expected` / `pretty_printed_received` match options used to
  be required to drop structural `\n`-leading whitespace from a single side
  in `:collapse` contexts. Those flags are now no-ops for `:collapse`
  parents — both sides are filtered unconditionally. They still affect
  `:preserve` contexts.

== Where it runs

The filter lives in two parallel call sites because canon has two children
filtering paths:

* `Canon::Comparison::MarkupComparator.node_excluded?` —
  used by `XmlComparator` / `HtmlComparator` via the
  `ChildComparison` positional alignment path.

* `Canon::Comparison::XmlNodeComparison.node_excluded?` —
  used by `XmlNodeComparison.compare_nodes` and
  `HtmlComparator.compare_fragment_children`.

Both apply the same `:collapse` + `\n`-leading rule so the fix is symmetric
across alignment paths.

== Configuration

There is no configuration knob for this filter. It is unconditional once
the parent's whitespace class is `:collapse`. The way to opt out is to put
the parent element on the `preserve_whitespace_elements` list (so it
classifies as `:preserve` instead of `:collapse`), at which point standard
strict-whitespace rules apply.
